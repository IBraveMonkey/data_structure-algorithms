A single commit should solve **exactly one** problem or implement one logical change.
- **Bad Practice**: A single massive commit affecting 50 files that fixes a bug, updates a library, and changes the UI layout.
- **Good Practice**: Three separate, small commits.
> [!TIP]
> Atomic commits are significantly easier to review, faster to revert if something breaks, and much easier to "cherry-pick" into other branches.

## ğŸ—„ï¸ 3. Repository Hygiene (.gitignore)
Maintain a clean repository by never committing unnecessary or sensitive files:
- **Dependencies**: Folders like `node_modules` or `vendor`.
- **Secrets**: Files containing API keys or environment variables (`.env`).
- **IDE Metadata**: Local configuration files from VS Code, IntelliJ, etc.
- **Build Artifacts**: Compiled binaries (`.exe`, `.log`).

## ğŸš€ 4. Professional Pro-Tips
- **Push Early, Push Often**: Do not hoard your code locally for a week. Pushing frequently allows colleagues to see your progress and helps avoid massive merge conflicts later.
- **Self-Review**: Before finalizing a commit or opening a Pull Request, run `git diff`. This is the best time to catch leftover `fmt.Println` debug statements or commented-out code.
- **Commit Squashing**: If you have a series of tiny commits like "fix typo" or "working...", use an interactive rebase (`rebase -i`) to "squash" them into a single, high-quality commit before merging into the main branch.
